# AstraHub 星链 · 更新日志

本文件记录 AstraHub 星链插件各版本的功能更新与问题修复。

## v2.4.5

本次更新聚焦友链相关页面的响应速度与交互流畅度。

### ✨ 性能与体验优化

#### 1. 友链星球卡片虚拟滚动改固定行高（根治越滑越卡）

- **问题背景**：友链星球列表此前用动态行高的虚拟滚动，每次滚动都要实测每行真实高度、维护前缀和偏移表（O(n) 重算），加载的卡片越多，每帧开销越大，往下滑越来越卡。
- **现在**：改为固定行高方案，行偏移用乘法 O(1) 直接算出，不再实测 DOM、不维护偏移表，移除了 ResizeObserver 行实测。无论累计加载多少条，DOM 节点数与每帧开销恒定，数万条友链滑到底也流畅。

#### 2. 审核「通过」改为局部即时响应

- **问题背景**：友链审核页点「通过」会触发整页重新加载（全量拉取收件箱 + 发件箱并同步），出现整页 loading，体验割裂。
- **现在**：与「撤回」一致，审核后直接用服务端回传的权威状态就地更新该条卡片，不再整页刷新，点击即响应。

#### 3. 数据响应速度从 3-4 秒提升到毫秒级

- 通过把搜索、关系筛选下沉服务端（数据库索引检索）+ 列表虚拟滚动 + 审核局部更新等一系列优化，友链数据的加载与交互响应从过去的 3-4 秒缩短到毫秒级。

### ⚙️ 部署须知（管理员）

- 本次为前端 Console 改动（`PlanetLinksPanel.vue`、`FriendInvitationManager.vue`），插件需重新构建安装。
- 虚拟滚动依赖固定行高，如后续自定义卡片高度，需同步对应行高常量。

## v2.4.4

本次更新修复友链邀请通过后，本地友链卡片在主题 links 页不显示的问题。

### 🐛 问题修复

#### 1. 友链分组名在本站不存在时卡片不显示

- **问题背景**：友链邀请的分组名（`linkGroupName`）是跨站共享字段，存的是「某一方本地」分组的标识，在审核方没选分组时还会回退成审核方的默认分组。该值在选它的那一方有效，到了另一方往往是本站不存在的分组名。官方 LinkFinder 在分组时，对「分组名非空但本站无此分组」的友链会落入孤儿桶被静默丢弃，导致 reconcile 生成的友链卡片在主题 links 页不展示。
- **修复**：reconcile 建本地友链前，先校验分组名是否真实存在于本站 LinkGroup。命中则原样保留；本站不存在（或查询异常）则降级为「未分组」（写入 null），保证卡片一定展示，不再被静默丢弃。

### ⚙️ 部署须知（管理员）

- 本次为插件服务端 Java 改动（`AstraHubFriendLinkReconcileService`），插件需重新构建安装。

## v2.4.3

本次更新聚焦「星链资讯（NewsHub）」左侧 RSS 来源列表的性能与可达性，把固定上限的来源列表改造为虚拟滚动 + 游标下滑加载，支撑数万级 RSS 源规模。

### ✨ 新增与增强

#### 1. RSS 来源列表「全量可达」（游标下滑加载）

- **问题背景**：星链资讯面板左侧的「RSS 列表」此前固定只取前 80 个来源，且只在首屏请求一次，没有任何下滑加载。当接入的 RSS 源数量超过 80 时，排在后面的站点（包括用户自己已注册的站点）在列表里看不到，只能靠搜索命中文章。
- **现在**：复用服务端 Discover 接口的游标分页（`nextCursor` / `hasMore`），滚动接近底部时自动拉取下一批，按 `sourceId` 去重追加，覆盖全量来源。不用再受 80 个上限限制。

#### 2. 来源列表虚拟滚动（根治长列表卡顿）

- 列表改为固定行高的虚拟滚动（窗口化）：按滚动位置只渲染可视区窗口（前后各留若干 overscan 行）。
- 无论累计加载多少条来源，DOM 节点数恒定，内存与滚动帧率稳定，可承载数万级来源不卡顿。
- 三个固定入口（稍后阅读 / 我的星系 / 全站 RSS）提为固定区，不参与滚动。

### ⚙️ 部署须知（管理员）

- 本次为前端 Console 改动（`NewsHubPanel.vue`），插件需重新构建安装。
- 虚拟滚动依赖固定行高，如后续自定义来源卡片高度，需同步对应行高常量。

## v2.4.2

本次更新聚焦「友链星球」的搜索、筛选与关系展示的彻底重构，把过去压在前端的逻辑全部下沉到服务端，支撑数万级友链规模，同时修复了多处关系展示与同步问题。

### ✨ 新增与增强

#### 1. 搜索框「全网可搜」（下沉服务端）

- **问题背景**：旧版搜索框只能在「已经滚动加载出来」的卡片里做本地过滤，没滑到的数据搜不到，数据量一大基本失效。
- **现在**：搜索关键词直接发给服务端，由数据库 `pg_trgm` 模糊索引在全量数据上检索，数万条也是毫秒级返回。不用再往下滑，输入即搜全网。
- 检索范围覆盖：站点标题、简介、URL、域名、标签。
- 输入防抖 300ms，避免每敲一个字就请求一次。

#### 2. 关系筛选「全网可筛」（下沉服务端）

- 「互相关注 / 我已关注 / 待回关」三个筛选档位全部改由服务端计算并筛选，作用于全量数据而非仅已加载部分。
- 服务端按关系档位把卡片统一排序（互关 → 我已关注 → 对方关注我 → 可发起邀请 → 已发邀请 → 普通），关系卡片首屏即置顶可见，无需滑到底。
- 「全部」「收藏」两项说明：
  - **全部**：不带关系筛选，返回完整统一流。
  - **收藏**：因为收藏是用户插件本地配置（Hub 不掌握），保留在前端本地过滤，这是数据归属决定的，非冗余。

#### 3. 关系状态展示以服务端为唯一真相源

- 卡片上的关系标签（互相关注 / 我已关注 / 他已关注 / 可发起邀请 等）完全由服务端 `relationStatus` 给定，前端不再自行推断。
- 修复历史 bug：**对方没接入插件但已经是好友的站点，被错误标成「暂未接入」**。现在关系来自友链边表，与对方是否注册无关，判定与排序档位同源，标签和位置永远一致。

#### 4. 信息全局自动同步

- 用户更新自己的站点信息（名称、简介、头像、地址等）后，所有引用到该站点的友链信息会自动同步统一，无需各方手动改。
- 站点身份以 `site_id` 锚定，**换域名也不断链**：底层关系认的是站点编号而不是 URL，迁移域名后历史友链关系完整保留。

### 🐛 问题修复

#### 1. 筛选按钮点了没反应（关键修复）

- **现象**：点关系筛选按钮，三个筛选请求返回完全一样的全量数据。
- **根因**：插件代理层（`AstraHubPlanetLinksRouter`）转发请求给服务端时，采用参数白名单，只放行了 `size`/`cursor`/`tag`，**把 `relation`（关系筛选）和 `keyword`（搜索词）静默丢弃了**，服务端根本没收到筛选条件。
- **修复**：代理层补充转发 `keyword`、`relation` 两个参数，筛选与搜索即刻生效。

#### 2. 已关注友链误显示「凭据缺失」

- 修复未接入站点的占位卡片在凭据检查时被错误判定为「凭据缺失」，恢复为正确的「尚未注册」文案，不影响认领流程。

#### 3. 站点身份合并 / 影子站清理

- 修复同一站点因新旧域名或重复上报产生的「影子站点」「双卡」问题，合并为单一权威站点身份。

### ⚙️ 部署须知（管理员）

- 服务端启动时会自动建库结构（`pg_trgm` 扩展 + 检索列 + 索引），无需手动操作。
- 升级后建议在后台执行一次「重建友链物化表」，回填检索字段。
- 插件需重新构建安装（本次代理层为 Java 改动）。
